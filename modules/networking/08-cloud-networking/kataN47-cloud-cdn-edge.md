# Kata N47 — Cloud CDN & edge

> **Track:** Networking · **Module:** N8 Cloud networking foundations · **Prereqs:** N46, N45, N25 · **Time:** ~35 min
> **Tags:** `networking` `cdn` `cloud` `l7-application` `cloud-cdn` `latency` `waf` `gcp`

## Why it matters

Meridian Bank's mobile-banking app lives in GCP `asia-south1` (Mumbai). A user
in Chennai — 1,400 km away — waits for a TLS handshake to travel to Mumbai and
back (~25 ms RTT) before seeing a single byte. A CDN terminates that TLS at a
Chennai PoP in under 5 ms. For Northwind's e-commerce portal, a CDN means 10 TB
of product images served per month are absorbed by edge caches, not billed as
origin egress. CDN is also where WAF rules and DDoS scrubbing live in modern
cloud architectures — it is the first thing a CISO will ask about for any
internet-facing service.

## The mental model

### Without CDN vs with CDN (two round-trips become one local hop)

```
WITHOUT CDN:
  User (Chennai) ────── TCP + TLS ──────────────── Origin (Mumbai)
                        2 RTTs × ~25 ms = ~50 ms overhead before first byte

WITH CDN PoP in Chennai:
  User (Chennai) ── TCP + TLS ─► PoP (Chennai)  ──── on cache miss ──► Origin (Mumbai)
                    < 5 ms RTT        │ serves cache hits instantly
```

The TLS handshake terminates at the PoP. The user never pays a long-distance RTT
just to set up the session.

### Cache hit vs cache miss

- **Cache hit** — PoP has the response; serves immediately; origin never contacted.
- **Cache miss** — PoP fetches from origin, caches the response, serves the user.

What gets cached is controlled by **`Cache-Control`** headers (RFC 7234) your
origin emits:

```
# Static asset — CDN caches 24 hours, browser refreshes after 10 min
Cache-Control: public, max-age=600, s-maxage=86400

# Authenticated API response — CDN must NOT cache
Cache-Control: no-store, private
```

The single most dangerous CDN misconfiguration in FSI: a missing `no-store` on
an account-balance API. The CDN caches one user's response and serves it to the
next user who requests the same URL — a PCI-DSS data breach.

### Where security controls attach

```
  Internet
     │
  [DDoS scrubbing]   ← volumetric floods absorbed at the edge (Cloud Armor / Shield)
     │
  [WAF at the edge]  ← L7 inspection: OWASP rules, bot signatures
     │                 (TLS terminates here → WAF sees plaintext HTTP)
  [CDN PoP]          ← cache hits served; misses forwarded
     │
  [Cloud LB]         ← see N46
     │
  [Origin in VPC]    ← Meridian: 10.100.0.0/14 (GCP); 10.104.0.0/14 (AWS)
```

This layering is a **conceptual** model, not the literal packet path on any one
cloud. In GCP specifically, Cloud Armor (WAF) and Cloud CDN are both features
co-located on the **same Global LB backend** — they are not separate tiers the
packet traverses in sequence (see the worked example and table below).

### Signed URLs for private content

Not everything is public. A CDN **signed URL** is a time-limited HMAC-signed
URL generated by your app. The PoP validates the signature before serving — the
object is cached at the edge, but only a holder of a valid (unexpired) URL can
receive it. Meridian uses these for PDF statements; Northwind uses them for
distributor price lists. TTL should be ≤ 300–600 seconds for sensitive material.

## Worked example

### Meridian Bank — GCP Cloud CDN + Cloud Armor

```
mobile.meridian.example
        │
  [Global Anycast IP] ← assigned by GCP Global LB
        │
  [Cloud Armor policy] ← WAF: sqli-v33-stable + xss-v33-stable + IP deny list
        │
  [Cloud CDN PoP]  ← PoP in Chennai, Singapore, London …
        │
  [GKE / Cloud Run] ← asia-south1, 10.100.0.0/14
```

GCP Cloud CDN is attached to a Global LB backend service — you cannot have it
without the Global LB (see N46). Enable via:
```bash
gcloud compute backend-services update meridian-backend \
  --enable-cdn \
  --cache-mode=CACHE_ALL_STATIC \
  --global
```
With ~70 % of requests being versioned static assets (JS/CSS/images) cached for
24 hours, expect a cache-hit ratio of 85–90 %. Origin sees only 10–15 % of
traffic. Monthly egress from `asia-south1` at $0.085/GB: 10 TB without CDN =
$850; 1.5 TB with 85 % hit ratio = $127.50. (Verify current pricing at
cloud.google.com/cdn/pricing before quoting to a client.)

### Northwind FMCG — AWS CloudFront

Northwind's distributor portal origin is in its AWS VPC (`10.50.0.0/16`).
CloudFront stands alone — no ALB required as the entry point (unlike GCP). A
CloudFront distribution points directly at the origin and handles its own global
anycast network. AWS WAF attaches to the CloudFront distribution for L7 filtering.
Private distributor price lists are served as **CloudFront signed URLs** (valid
300 s); the app generates one at login time.

## Cloud / vendor mapping (when applicable)

| Concept | On-prem | GCP | AWS | Azure |
|---------|---------|-----|-----|-------|
| CDN / edge cache | Akamai / Fastly / Varnish | **Cloud CDN** (must attach to Global LB) | **CloudFront** (standalone distribution) | Azure CDN / Azure Front Door |
| WAF at edge | F5 / Palo Alto appliance | **Cloud Armor** (policy on Global LB) | **AWS WAF** (on CloudFront or ALB) | Azure WAF (Front Door or App Gateway) |
| DDoS scrubbing | ISP scrubbing center | **Cloud Armor** standard (automatic) | **AWS Shield Standard** (automatic) | Azure DDoS Protection |
| Private / signed content | Token-auth URL from app | **Cloud CDN signed URLs** (classic key: HMAC-SHA1) | **CloudFront signed URLs** or signed cookies | SAS tokens on Azure CDN |
| Global anycast entry | BGP anycast from DC edge | GCP Global LB anycast IP | CloudFront edge PoP network | Azure Front Door |

**Key structural difference:** GCP Cloud CDN is always coupled to a Global LB —
the LB is the entry point, CDN is a feature on its backend. AWS CloudFront is
independent — you can point it at S3, an ALB, or any HTTP endpoint. Azure Front
Door (Standard/Premium) bundles CDN + WAF + global LB into one service.

## Do it (the exercise)

### 1 — Inspect CDN headers in the wild [laptop]

```bash
curl -sI https://www.google.com | grep -iE "cache-control|age|x-cache|via"
```
Look for `Age: 3421` (seconds in cache), `x-cache: Hit`, or
`cf-cache-status: HIT`. Now compare a dynamic endpoint:
```bash
curl -sI https://httpbin.org/get | grep -i cache-control
```

For Meridian Bank, fill in the correct header and reason:

Expected: `public, s-maxage=86400` for versioned static; `no-store, private`

A Meridian API response carries:
```
Cache-Control: public, max-age=600
```
This endpoint returns the logged-in user's account balance. Identify: (a) what
the CDN will do, (b) the security impact, (c) the correct header.

## Talk to the IT/security head

**Ask:**
- "What is your cache-hit ratio, and do you have alerting when it drops?"
  A good answer: they know the number (target > 80 % for static workloads), it
  is visible in Cloud Monitoring or CloudFront metrics, and a sudden drop triggers
  an alert (often means a cache-key regression broke caching).

- "Which paths have `no-store` enforced, and is that tested in your CI pipeline?"
  A good answer: a header test runs on every deploy; origin *and* CDN are both
  checked. Red flag: "we think the app sets it" — untested; one misconfigured
  endpoint is a breach.

- "Can your origin be reached directly, bypassing the CDN/WAF?"
  A good answer: origin only accepts traffic from CDN/LB IP ranges (GCP: LB
  health-check ranges; AWS: CloudFront IP list in `ip-ranges.json`). Red flag:
  the origin has a public IP or public DNS name that bypasses the WAF entirely.

- "What is your CDN fallback if the CDN provider has an outage?"
  A good answer: a tested runbook switches DNS to serve directly from origin
  (latency degrades, but service stays up). Red flag: "the CDN never goes down"
  — Fastly (June 2021) and Akamai (July 2021) both went down hard.

**Red flags:** no CDN visibility; WAF on pilot rather than enforced; no
difference in `Cache-Control` between static and authenticated paths; signed
URLs with TTLs measured in hours, not minutes.

## Pitfalls & war stories

- **Caching authenticated content** — the top CDN failure in FSI. `Cache-Control:
  public` on a balance endpoint is a PCI-DSS finding and a live data leak.
- **Cache key collisions** — URL-only cache keys plus a response that varies by
  `Cookie` or `Authorization` header means stale data is served to the wrong
  user. Configure custom cache keys (GCP: cache key policy; AWS: CloudFront cache
  policy) to include or exclude the right headers.
- **WAF bypass via direct origin** — if origin has any public IP, an attacker
  skips the WAF. Restrict origin to CDN/LB source IPs, always.
- **Northwind M&A audit gap** — after the Eastfield Foods acquisition, Northwind
  had two CloudFront distributions with different WAF policies; the acquired
  entity's had no WAF at all. CDN/WAF configs must be part of M&A due diligence.
